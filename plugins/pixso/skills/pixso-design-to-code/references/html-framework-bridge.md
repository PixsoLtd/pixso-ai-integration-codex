# HTML Bridge to Target Frameworks

Use this reference when the target framework is not supported by `design_to_code`. Generate HTML first, then convert the HTML / CSS / assets to the target framework with a 1:1 structure.

The HTML bridge only creates the first target-framework draft; it is not the delivery endpoint. The converted target-framework output must return to the main workflow for resource handling, screenshot context, Phase A cleanup, Phase B cleanup, and verification.

## Inputs

- HTML, CSS, and assets generated by `design_to_code`.
- Visual context from `get_screenshot(guid)`.
- The user-specified target framework and project constraints.

## Rules

- Adapt only syntax, file organization, and asset paths.
- Preserve DOM hierarchy, text, style declarations, dimensions, spacing, colors, stacking, images, and responsive behavior.
- Reuse CSS as-is when possible; when rewriting is unavoidable, preserve selector semantics and visual output.
- Add only the minimum integration code required by the target framework, such as a component entry, export, or lifecycle hook.

## Forbidden

- Do not redesign layout, visuals, copy, or interactions.
- Do not add states, animations, data logic, component abstractions, or a design system.
- Do not reorder structure, merge nodes, or split nodes just to match target-framework habits.
- Do not introduce unrequested dependencies, UI libraries, or CSS frameworks.
- Do not treat HTML bridge conversion as `refine_generated_code` or cleanup refactoring.
- Do not deliver immediately after conversion while skipping `references/cleanup.md`.

## Completion check

- The HTML bridge source is traceable.
- Target-framework code preserves the HTML source structure and styles 1:1.
- The target-framework output continued through `references/cleanup.md`.
- Any difference that could not be preserved 1:1 is explained with a reason.
